## Prompt

```text
 Create a comprehensive project constitution that governs the entire
5-phase lifecycle of this hackathon project.

This constitution must align with:
- Hackathon evaluation standards
- Spec-Kit Plus philosophy
- Spec-Driven Development
- Agentic, AI-native workflows using Claude Code

====================
PROJECT SCOPE
====================

This constitution applies to ALL phases (Phase 1 through Phase 5)
of the project and must remain stable across the entire hackathon.

Phase-specific details will live in individual specifications,
but this constitution defines the permanent rules, principles,
and governance model.

====================
CORE PHILOSOPHY
====================

This project follows Spec-Driven Vibe-Coding:

- Specifications are the primary artifact
- Code is a derived artifact generated by AI
- Human responsibility is architectural thinking,
  constraint definition, validation, and review

The constitution must emphasize that:
- Writing specs is more important than writing code
- AI is a collaborator, not an autopilot
- Correctness, clarity, and intent matter more than speed

====================
DEVELOPMENT PRINCIPLES
====================

Define principles covering:

1. Spec-First Development
   - No code before specs
   - Specs define behavior, not implementation
   - Specs are versioned and immutable once approved

2. Phase Isolation with Forward Compatibility
   - Each phase must be independently runnable
   - Later phases may extend but must not break earlier phases
   - All phases live in a single repository with clear boundaries

3. Agentic Discipline
   - The AI agent must follow explicit workflows
   - No skipped steps (spec ‚Üí plan ‚Üí tasks ‚Üí implement)
   - No unapproved feature introduction

4. Human-in-the-Loop Governance
   - Humans approve specs and transitions between phases
   - Humans validate correctness and intent
   - Humans remain accountable for outcomes

====================
ARCHITECTURAL GOVERNANCE
====================

The constitution must establish rules for:

- Clean project structure
- Separation of concerns
- Modularity and extensibility
- Deterministic behavior
- Explicit error handling

It must state that:
- Early phases may be simple, but never sloppy
- Design decisions must be documented
- Trade-offs must be explicit

====================
AI USAGE RULES
====================

Clearly define how AI is used:

- AI generates specs, plans, tasks, and code
- AI must explain reasoning between phases
- AI must not hallucinate requirements
- AI must not bypass guardrails

AI output must always be:
- Reviewable
- Traceable to a spec
- Aligned with acceptance criteria

====================
QUALITY STANDARDS
====================

Define non-negotiable quality bars:

- Readability over cleverness
- Explicit behavior over implicit assumptions
- Predictable CLI interfaces
- Consistent naming and structure
- Minimal but sufficient documentation

====================
DOCUMENTATION REQUIREMENTS
====================

The constitution must require:

- A README explaining phases and usage
- A CLAUDE.md explaining agent usage
- Maintained spec history
- Architecture and decision records when needed

====================
HACKATHON ALIGNMENT
====================

Explicitly state that the goal is to demonstrate:

- Spec-driven thinking
- AI-native development maturity
- Architectural foresight
- Clean evolution across phases
- Professional engineering judgment

====================
TONE & FORMAT
====================

- Write in clear, authoritative language
- Avoid marketing fluff
- Sound like a real engineering constitution
- Use numbered sections and bullet points
- Keep it concise but complete

====================
OUTPUT
====================

Produce a single file named:

constitution.md

This constitution must be suitable for evaluation by judges
as evidence of disciplined, AI-native software engineering.
```
